# Java 线程与内存分析

## 概念

### 线程锁定

- 线程在唤醒状态，但因争用对象锁过程中没有获得对象锁，而不能执行
  - 如进入 synchronized 同步块
- 若两个线程互相争用对方【已经持有的】对象锁而锁定，则导致【线程死锁】

### 线程泄漏

- 泄漏的线程占用线程池资源
- 泄漏线程虚拟机栈栈帧引用的临时对象不可回收，占用堆内存

#### loseWakeup 泄漏

- 线程 wait 挂起后无法被唤醒
  - 线程获得对象锁之后执行了 wait 挂起，但没有其他线程通过这个对象执行 notify，导致该线程无法被唤醒
  - 典型如生产/消费线程模型中，生产线程判断挂起时，因内存模型原因，主存中满足挂起条件，CPU 中实际不满足挂起条件，导致线程错误挂起。如果后续因主存数据条件不满足唤醒，则造成 loseWakeUp
- 要求设计有保障的线程唤醒机制
  - 内存屏障保证主/缓存一致，避免错误挂起
  - 如果怕设计不够精巧，可以使用限时 wait 替代不限时 wait，在一定时间后可以自我唤醒，判断是否需要继续挂起

#### 死锁泄漏

- 线程死锁之后，虚拟机外上层应用超时释放连接，但虚拟机内两个死锁线程一直存在
- 死锁的线程除了占用堆内存，还会卡死所有其他申请对象锁的线程

### 线程切换损耗

- 当线程数大于 CPU 数量时，单个 CPU 按时间分片执行多个线程，在一个线程的分片时间执行完成之后，会切换到另外一个线程继续执行
- 切换线程时需要保存切换前线程上下文，载入切换后线程上下文，造成性能损耗
  - OLTP 业务场景中，系统瓶颈常在数据传输（比如引擎智审场景），该部分影响不大
  - OLAP 业务场景中，系统瓶颈常在 CPU（比如引擎 G 端监管场景），此时需要考虑
- OLAP 场景中从整体计算性能考虑，通过控制【运行线程数量】，减少线程切换，让 CPU 处于【尽量满载、不要超载】的最佳状态
  - 查看 CPU 百分比，查看 load Average CPU 负载走势

### GC Root

- GC Root 是【引用类型】的一种分类，常见如下 GC Root
  - 虚拟机栈栈帧中【本地变量表】条目，（即执行中的方法的局部变量）
  - 本地方法栈栈帧变量条目
  - 方法区中类静态 field
- GC Root 可达，指被 GC Root 直接引用，或者被其引用的对象递归引用。除了 PhantomReference、WeakReference、SoftReference，GC Root 可达对象不会被回收
- 当虚拟机栈帧弹出/本地方法栈栈帧弹出（方法执行完成），静态类属性赋值改变时，对象 GC Root 被移除，此时对象仍然在堆中，但下次 GC 即会回收该对象

### 内存泄漏

- 创建的对象被放入了【静态变量】递归持有的对象 field 中形成 GC Root。程序执行完成后，这部分对象失去作用，但因程序缺陷，未能手动移除 GC Root，如此形成内存泄漏
  - 注意区别于高并发线程栈帧持有的对象
- 积累性的内存泄漏持续占用堆内存
  - 当老代满时执行 FullGC，GC 后空余内存仍然不满足分配，或者 GC 回收的内存很少，回收时间占比较多，则会 OOM
  - 若没有触发 OOM，则会更频繁触发 FullGC，导致 CPU 虚耗

### 内存资源评估

- 更多的资源于程序来说总是更好的，但过多的资源造成资源浪费，需要能评估内存合理性
- 新代中生成的临时对象尽量不要进入老代，老代仅存储长时间缓存类数据
  - 如果发现 minorGC 比较频繁，且 fullGC 之后老代内存大幅减少，说明大量这种类型的对象进入老代，此时应该扩大 eden 减少 minorGC 次数，扩大 survivor 能容纳 GC 后的存活对象
  - 如果发现【多次】GC 后老代存活的对象较少，则考虑内存富余，减少老代容量，节约内存

## 进程查看

- ps（Linux），taskList（Windows）等命令能查看启动命令行中包含 java 关键字的 Java 进程
- jps 可查看所有进程

```bash
# 打印【进程号】，【完整路径执行文件名/Main 函数类完全限定名】，【option】，【参数】
jps -lmv
```


- 图形化
- 查看线程状态分布

#### GC 日志开启

```bash
# VM 增加启动参数
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCCause -Xloggc:gclog-%t.log
```

```bash

```

## 实战经验 - 内存泄漏

### 问题

- 业务模型驱动开发平台群中反应该产品每隔一两个星期平台会异常，相关系统都不能正常使用，每次重启解决
- 每晚预审业务场景中出现
- 持续 CPU 满载，但业务线程都卡死
- arthas 连不上，jstack 连不上

### 第一反应：看看堆内存情况

- 要到服务器连接方式，登录后【jmap -heap 】发现老代使用率保持 99%
- 推测是在【频繁 GC】。因为 GC 在 STW，业务线程会全卡死执行不了，【导致 arthas，jstack 连不进去】

### 第二反应：看是为什么占了这么多

- 【jmap -histo:live】发现 AtomicInteger，Shardingsphere.Process 等类有上千万个对象
- 推测可能存在【线程或内存泄漏】
- 让现场同事 dump 内存进一步分析

### dump 到手后，分析内存

- 查 Process 对象引用，发现都被一个 ProcessRegistry 类型的【静态 field】引用，且【不存在栈帧引用】，则判断为此处【内存泄漏】
- 定位 ShardingSphere 工具包问题。去该工具 github 的 issue 搜索 memory leak，找到 5.4.0 版本有相同 issue 描述，遂询问产品组使用的版本，匹配确认该内存泄漏问题

### 问题复盘

- 业务模型驱动使用了 ShardingSphere 做分布式 JDBC 连接，这个工具的 5.4.0 版本存在内存泄漏 bug
- 每晚预审因交易笔数多，导致该业务中泄漏速度快，容易触发达到崩溃瓶颈
- 当达到瓶颈时，不停地 FullGC 导致 STW，普通 java 线程无法执行，卡死了包括 arthas 等诊断工具线程
- 因每次 GC 后回收的少量内存能达到分配条件，没有触发 OOM
- 不停地 FullGC 导致 CPU 满载

## 实战经验 - 内存资源评估

### 问题

- 反应 jvm 占用近 8 个 G 的操作系统内存，快占满了，虽然系统正常运行，【但感觉有问题】

### 第一反应：通常应用不会实际占这么多内存，先看看 GC 后的堆内存情况

- 【jmap -heap 】发现 eden 300M，old 7G，GC 后 old 170M

### 分析内存配置过大

- Eden 只有 300，说明没有大量【并发占用内存】的情况
- Old 7G，且回收后 170M，典型的积累性新代临时对象进老代的情况
- 推断内存富余

### 让实施人员将 -Xmx 调整到 4G 即可

- 根据 Eden 的分配判断，实际该场景 1G 内存应该够，只是 4G 保险
